# 0. About this Document

## 0.1 目的

本ドキュメントは、ジムグリット コーポレートサイトにおいて、検索ユーザーへ価値ある情報を届け、理解、信頼、比較、問い合わせへつなげるためのSEO方針を定義する。

SEOを検索順位の操作として扱わず、ユーザーの疑問に答える情報、ジムグリットの専門性、実際のProject（制作・支援実績）、会社としての信頼を継続的に蓄積する活動として整理する。

## 0.2 責務

本ドキュメントでは、次を扱う。

- SEOに対する基本姿勢
- Search Intentの分類と対応するページ
- 検索を起点としたContent Portfolio
- Internal Linkingの戦略
- Brand Searchに対する情報整備
- E-E-A-Tを支える情報と運用
- AI Searchを考慮した情報資産
- 評価、更新、改善、Version管理

## 0.3 他のStandardとの違い

- Web Strategyは、サイトの目的、ページ構成、事業導線を定義する。
- Content Guideは、ページ内で何を、どの順番で伝えるかを定義する。
- Development Guideは、index、Canonical、Metadata、Structured Data、Sitemapなど技術的SEOを定義する。
- SEO Guideは、どの検索ユーザーへ、どのテーマとページで価値を届け、どのように資産として育てるかを定義する。

本ドキュメントでは、技術設定、画面表現、実装方法、完成原稿、文章校正ルールを定義しない。技術的SEOは `04_Development_Guide.md`、情報構造と文章品質は `05_Content_Guide.md` を参照する。

## 0.4 適用原則

検索需要があることだけを理由に、ジムグリットの事業、専門性、ユーザーへの提供価値と関係のないテーマを扱わない。

判断は、次の順で行う。

1. ユーザーの課題解決に役立つか。
2. ジムグリットが責任を持って伝えられるか。
3. 一次情報、経験、専門性、独自の判断を加えられるか。
4. Service、Project、Companyの理解へ自然につながるか。
5. 継続して更新・改善できるか。

# 1. SEO Philosophy

## 1.1 SEOは資産である

SEOは、短期的な集客施策ではなく、ユーザーの疑問に答える情報とブランドへの信頼を蓄積する活動である。

有用なServiceページ、具体的なProject、独自の知見、正確な会社情報は、公開後も検索、Brand Search、営業、採用、AI Searchなど複数の接点で利用される。記事本数ではなく、長く参照できる情報資産が増えているかを重視する。

## 1.2 検索順位だけを目的にしない

順位は成果を観察する一つの指標であり、SEOの最終目的ではない。上位表示しても、検索意図に答えず、ブランドへの理解や適切な行動につながらないページは成功とみなさない。

評価では、次を合わせて見る。

- 適切な検索ユーザーに発見されているか。
- ページ内で疑問が解決されているか。
- 関連するService、Project、Companyへ理解が進んでいるか。
- Brand Searchと再訪が増えているか。
- 問い合わせ、相談、採用など事業上の行動へつながっているか。
- 公開後も正確性と価値を維持できているか。

## 1.3 ユーザー課題を解決する

SEO ContentはKeywordを含めるためではなく、検索の背景にある課題、疑問、比較、決定を助けるために作る。

検索ユーザーがページを読んだ後、必要な判断材料を得て、同じ疑問で再検索しなくてもよい状態を目指す。答えが案件、条件、状況によって異なる場合は、一つの結論に見せず、判断軸と相談が必要な範囲を示す。

## 1.4 ブランド価値を積み上げる

検索流入を増やすために、ブランドの専門領域を無制限に広げない。ジムグリットが実際に経験し、支援し、責任を持てるテーマを継続して扱う。

Contentを通じて、次の印象を積み上げる。

- 現場と事業を理解している。
- 複雑な課題を整理して説明できる。
- 実際の経験と根拠に基づいている。
- 成果だけでなく、条件や限界も誠実に伝える。
- 情報を継続的に更新している。

## 1.5 People-first

SEOのためだけにページを量産せず、人が直接訪れても役立つContentを作る。[Google Search CentralのPeople-first Content指針](https://developers.google.com/search/docs/fundamentals/creating-helpful-content?hl=ja)を参照し、検索エンジン向けの形式より、独自性、完全性、正確性、利用後の満足を優先する。

次の状態を避ける。

- 検索量だけで選んだ、事業と関係の薄いテーマ
- 他Siteの要約や言い換えだけで構成されたContent
- 同じSearch Intentに対する類似ページの量産
- 内容を変えず、更新日だけを新しく見せる運用
- 文字数、Keyword出現回数、記事本数を目的にした制作
- AIや外部Writerへ大量委託し、個別の確認と専門性が失われた状態

# 2. Search Intent

## 2.1 Search Intentの考え方

同じKeywordでも、ユーザーが知りたいこと、比較したいこと、行動したいことは異なる。Keyword単体ではなく、検索の背景、必要な答え、検討段階、次の行動を一組として整理する。

各ページにはPrimary Intentを一つ定める。複数のIntentを扱う場合も、主目的を曖昧にしない。

## 2.2 情報収集

**ユーザーの状態**  
課題やテーマについて知りたいが、依頼先や具体的な行動はまだ決めていない。

**主な疑問**

- これは何か。
- なぜ必要か。
- どのような考え方や選択肢があるか。
- 自分の状況では何を確認すべきか。

**対応するページ**

- Column / Guide
- 課題別の解説
- Serviceの導入説明
- FAQ
- 用語やProcessの解説

**必要な価値**

- 結論と基本的な理解
- 判断軸と注意点
- ジムグリットの経験から得た補足
- 次に読むべき関連情報

情報収集段階でContactを迫らず、理解を深めるService、Project、FAQへ自然につなぐ。

## 2.3 比較検討

**ユーザーの状態**  
課題と解決方法を理解し、Service、支援会社、進め方、条件を比較している。

**主な疑問**

- 何をどこまで依頼できるか。
- 他の選択肢と何が違うか。
- 自分に適しているか。
- どのように進み、どのような実績があるか。

**対応するページ**

- Service詳細
- Project / Case Study
- 支援Process
- FAQ
- Company / About

**必要な価値**

- 対象、提供範囲、条件、対象外
- 特徴ではなく選択に必要な違い
- 実際のProjectと支援範囲
- Team、会社、責任の所在
- 相談前に確認できる情報

比較対象を不当に下げず、ジムグリットが適する条件と適さない条件を誠実に示す。

## 2.4 問い合わせ・行動

**ユーザーの状態**  
相談、依頼、連絡を検討し、具体的な方法と行動後の流れを確認している。

**主な疑問**

- 何を相談できるか。
- 誰が対応するか。
- 何を準備すればよいか。
- 送信後、いつ、どのように連絡が来るか。

**対応するページ**

- Contact
- Service詳細
- FAQ
- Company Profile
- 必要に応じて相談Flow

**必要な価値**

- 問い合わせ対象と窓口
- 入力前に必要な条件
- 回答方法と目安
- Privacyと安心材料
- 送信後の見通し

## 2.5 指名検索

**ユーザーの状態**  
ジムグリット、代表者、Service、Projectをすでに知り、公式情報や信頼性を確認している。

**主な疑問**

- 公式Siteはどれか。
- どのような会社で、誰が運営しているか。
- 何を提供し、どのような実績があるか。
- 安心して連絡できるか。

**対応するページ**

- Home
- About / Company
- Service
- Project
- News
- Contact

**必要な価値**

- 正式な会社情報とブランド表記
- 代表者と事業内容
- 最新のServiceとProject
- 公式なNewsと外部発信への接続
- 明確なContact

## 2.6 Intent Map

主要Keyword群は、次の項目で管理する。

| 項目 | 内容 |
| --- | --- |
| Query / Theme | 実際の検索語と、それらを束ねるテーマ |
| Primary Intent | 情報収集、比較検討、問い合わせ、指名検索 |
| User Question | 検索の背景にある具体的な疑問 |
| Target Page | 疑問へ最もよく答える一つの主要ページ |
| Supporting Page | 理解やEvidenceを補う関連ページ |
| Next Step | 読了後に自然な情報または行動 |
| Evidence | 経験、Project、数字、人物、出典など |
| Owner | 正確性と更新を担う担当者 |

同じIntentを複数ページで競合させない。既存ページで十分に答えられる場合は新規作成より改善を優先し、重複するページは統合または役割を分ける。

# 3. Content Strategy

## 3.1 基本方針

検索ユーザーの疑問を解決しながら、ジムグリットの実際の経験、専門性、Service、Project、Companyへの理解を深める。

「検索されていること」と「ジムグリットが伝える資格と価値があること」が重なる領域を優先する。

## 3.2 Theme領域

SEO Contentは、次のTheme領域から選ぶ。

| Theme領域 | 役割 | 主な情報源 |
| --- | --- | --- |
| User Problem | Targetが抱える課題を整理する | 相談、現場観察、問い合わせ、営業知見 |
| Service Knowledge | 解決方法、判断軸、Processを伝える | 実務、専門知識、支援方法 |
| Project Insight | 実際の課題、判断、支援、結果を示す | Project記録、担当者、Client承認情報 |
| Company Expertise | 人物、事業姿勢、専門領域を示す | 代表者、Team、沿革、活動実績 |
| FAQ | 繰り返し生じる具体的な疑問へ答える | 問い合わせ、商談、運用記録 |
| News / Primary Information | 会社や事業の最新事実を公式に示す | 社内決定、発表、Event、提携、更新 |

事業との関係が説明できないTheme、一次情報を加えられないTheme、継続確認できないThemeは扱わない。

## 3.3 Content Portfolio

単発記事を増やすのではなく、役割の異なるページを組み合わせてTopicを立体的に伝える。

| Content Type | SEO上の役割 |
| --- | --- |
| Home | ブランドと事業領域の中心を示す |
| Service | 比較・相談に必要な提供価値と範囲を示す |
| Project | ExperienceとEvidenceを示す |
| About / Company | 運営主体、人物、専門領域、Trustを示す |
| Column / Guide | 情報収集段階の具体的な疑問へ答える |
| FAQ | 検討・行動前の細かな疑問を補う |
| News | 公式な一次情報と活動の継続性を示す |
| Contact | 行動Intentを受け止める |

一つのTopicについて、CoreとなるServiceまたはGuide、EvidenceとなるProject、補足となるFAQやColumnを関連付ける。

## 3.4 Content企画の優先順位

新規作成または改善の優先度は、次の観点で判断する。

1. 主要Targetの具体的な疑問か。
2. Serviceまたは事業目標との関係が強いか。
3. ジムグリット独自の経験、Project、判断を加えられるか。
4. 既存ページで十分に答えられていないか。
5. 比較、信頼、問い合わせなど次の理解へつながるか。
6. 正確性を維持するOwnerと情報源があるか。
7. 一時的な流行ではなく、継続的な価値があるか。

検索Volumeだけで優先順位を決めない。小さな需要でも、重要なTargetの意思決定に直結するThemeは優先できる。

## 3.5 Original Value

他Siteにもある一般論だけで終わらせず、次のいずれかを加える。

- 実際のProjectで得た知見
- 現場の写真、Process、判断理由
- 独自に整理した比較軸やFramework
- 代表者または担当者の専門的見解
- 自社で確認したDataや傾向
- 失敗、制約、適さない条件を含む実務的な説明
- 最新の会社・Serviceに関する一次情報

外部情報を利用する場合は、出典を明確にし、その要約だけでなくジムグリットとしての解釈または実務上の価値を加える。

## 3.6 継続更新

公開後は、順位だけでなく、検索意図への適合、情報の正確性、関連ページへの移動、事業との関係を確認する。

更新時の選択肢は、次のとおりとする。

- **Improve**: 答え、Evidence、構成、関連情報を補強する。
- **Merge**: 同じIntentの類似ページを一つへ統合する。
- **Separate**: 一つのページに異なるIntentが混在する場合、役割を分ける。
- **Archive**: 現在価値がなく、誤解を生む情報を公開終了する。
- **Maintain**: 現在も十分に有用で正確な場合、不要な変更をしない。

鮮度を演出するためだけに更新日を変更せず、内容に実質的な変更がある場合だけ更新として扱う。

## 3.7 SEO Brief

新規Contentまたは大きな改善では、最低限次を整理する。

- Target User
- Primary Search Intent
- User Question
- Target Theme / Query群
- ページが提供する結論と独自価値
- 必要なExperience、Evidence、Source
- 関連するService、Project、Company、FAQ
- 次に読む情報または行動
- Author / Reviewer
- Ownerと更新Trigger
- 成功を確認する指標

本文の情報順序とWritingはContent Guideに従う。

# 4. Internal Linking

## 4.1 目的

Internal Linkingは、ユーザーが一つの疑問から次の理解へ移るための案内であり、サイト内の情報関係と重要度を示す仕組みである。

Link数を増やすことを目的にせず、現在のページだけでは不足する情報、主張を裏付けるEvidence、次の判断や行動を補うページへつなぐ。

## 4.2 基本構造

```text
Home
├── Service
│   ├── Project
│   ├── Column / Guide
│   ├── FAQ
│   └── Contact
├── Project
│   ├── Related Service
│   └── Contact
├── About / Company
└── News
```

ColumnやFAQを入口として孤立させず、関連するService、Project、Company、Contactへ理解が進む構造にする。

## 4.3 Linkの役割

| Linkの役割 | 接続例 |
| --- | --- |
| 定義・補足 | Columnから関連GuideやFAQへ |
| 解決方法 | 課題解説からServiceへ |
| Evidence | ServiceからProjectへ |
| 専門性・責任 | ColumnやProjectからAuthor、About、Companyへ |
| 次の検討 | Projectから関連ServiceやFAQへ |
| 行動 | Service、Project、FAQからContactへ |

## 4.4 重要ページ

- 主要ServiceとCompany情報は、Homeと関連Contentから到達できるようにする。
- 重要ページをFooterやSitemapだけに依存させない。
- 高い検索流入があるページから、無関係な売上ページへ機械的にLinkしない。
- 関連する複数ページから自然に参照される状態を作る。
- 新規ページ公開時は、既存のどのページからLinkするかを同時に決める。

## 4.5 Anchor Text

Anchor Textは、移動先の内容を具体的かつ簡潔に伝える。「こちら」「詳しく見る」だけに依存せず、文脈がなくても移動先を予測できる名称を使う。

同じページへのAnchor TextをKeywordで不自然に統一せず、文章の文脈に合わせる。検索語を詰め込むための長いAnchor Textや、同じLinkの過剰な反復を避ける。

[Google Search CentralのLink Best Practices](https://developers.google.com/search/docs/crawling-indexing/links-crawlable)も参照し、ユーザーと検索システムの双方が関係を理解できるLinkにする。技術的なLink出力はDevelopment Guideで管理する。

## 4.6 孤立ページ

公開対象の重要ページには、少なくとも一つ以上の文脈上関連する内部Linkを設ける。次を定期的に確認する。

- どのページからも参照されない孤立ページがないか。
- 廃止したページへのLinkが残っていないか。
- 役割が重複するページ同士が競合していないか。
- ServiceとProject、ColumnとServiceの接続が切れていないか。
- Link先の内容がAnchor Textと一致しているか。
- Contact以外の次の理解手段も用意されているか。

# 5. Brand SEO

## 5.1 目的

Brand SEOは、ジムグリット、代表者、Service、Projectを知ったユーザーが検索したときに、正確で一貫した公式情報へ到達し、会社を信頼して次の判断へ進める状態をつくることである。

指名検索を単なるHomeへの流入と考えず、評判確認、実在性確認、Service確認、採用、取材、問い合わせなど複数の目的を持つ検索として扱う。

## 5.2 会社情報

次の情報を公式Site内で明確かつ一貫して示す。

- 正式会社名とブランド名
- 所在地と連絡先
- 設立、法人情報、事業内容
- 代表者
- 主要ServiceとBrand
- 主要Projectと実績
- 公式SNSや外部発信
- Privacy PolicyとContact

表記、住所、役職、Logo、Profile、URLが、Site内と管理する公式外部Profileで矛盾しないようにする。

## 5.3 代表者情報

- 正式な氏名と役職を統一する。
- 担当領域、経歴、Experience、専門性を事業との関係が分かる形で示す。
- 代表Messageだけでなく、Project、Column、登壇、取材など実際の活動へ接続する。
- 公開する経歴、資格、実績は確認可能な事実に限定する。
- Authorとして発信する場合は、Profileと専門領域をContentから確認できるようにする。

## 5.4 事業内容とService

- Companyの事業内容と個別Serviceの関係を明確にする。
- Service名、対象、提供範囲を一貫して表記する。
- 終了したServiceを現在も提供中に見せない。
- Brand、Service、Projectを混同させない。
- 指名検索ユーザーが、提供内容とContactへ短い導線で到達できるようにする。

## 5.5 実績とProject

- 実際の支援範囲、判断、Process、結果を示す。
- Client名、Logo、写真、成果は掲載許可を確認する。
- 事例と自社Serviceを関連付ける。
- 数字には期間、対象、条件、確認元を示す。
- 過去の実績を現在の能力として誤認させない。
- Project一覧だけでなく、代表的なProjectを詳細に説明する。

## 5.6 Brand Searchの確認

次のようなQueryで、ユーザーが必要な公式情報へ到達できるかを確認する。

- ジムグリット
- 株式会社ジムグリット
- ジムグリット 代表
- ジムグリット Service名
- ジムグリット Project名
- ジムグリット 会社概要
- ジムグリット 問い合わせ

検索結果に誤情報、古い情報、別組織との混同がある場合は、まず公式Siteと管理可能な公式Profileの情報を整える。第三者評価を操作せず、必要に応じて正確な一次情報を公開する。

## 5.7 Reputation

- 実在するClient、Partner、Media、Eventなどからの言及を正確に記録する。
- 第三者の評価と自社の主張を区別する。
- Reviewや推薦を依頼する場合も、内容を誘導または捏造しない。
- Negativeな情報を隠すためのページ量産を行わない。
- 誤りがある場合は、訂正可能な公式情報を明確に公開する。

# 6. E-E-A-T

## 6.1 E-E-A-Tの位置付け

E-E-A-Tは、Experience、Expertise、Authoritativeness、Trustworthinessの観点からContentとSiteの信頼性を自己評価するための考え方として扱う。

E-E-A-T自体は単一のランキング要因ではない。GoogleはTrustを最も重要な中心要素と説明しているため、要素を形式的に追加するのではなく、ユーザーが「誰が、どの経験と根拠に基づき、なぜこの情報を伝えているか」を確認できる状態を目指す。詳細は[Google Search CentralのE-E-A-T解説](https://developers.google.com/search/docs/fundamentals/creating-helpful-content?hl=ja)を参照する。

## 6.2 Experience

実際に行ったこと、関わったこと、現場で得た知見を示す。

- Projectの背景、課題、担当範囲、Process、判断
- 現場、人物、成果物の承認済み写真
- 支援中に確認した課題と学び
- 成果だけでなく、制約、失敗、改善
- 代表者や担当者が実際に経験した内容

一般論へ後付けでProject名を添えるのではなく、経験からしか得られない具体性をContentの中心にする。

## 6.3 Expertise

正確な知識と、複雑な内容を判断可能な形へ整理する力を示す。

- AuthorまたはReviewerの氏名、役割、専門領域
- 経歴、資格、実績のうち、Themeに関係する情報
- 判断の基準、根拠、Source
- 専門用語を正確かつ分かりやすく説明するContent
- 重要な主張に対する専門家のReview

健康、安全、経済的判断など影響の大きいThemeでは、適切な資格・専門性を持つ人の確認と、信頼できるSourceを重視する。

## 6.4 Authoritativeness

特定領域について継続的に価値を提供し、社内外から参照される状態をつくる。

- Site全体のThemeと専門領域を一貫させる。
- Service、Project、Column、代表者情報を相互に関連付ける。
- 独自のProject、調査、Framework、知見を公開する。
- Client、Partner、Media、業界団体など、確認可能な関係を示す。
- 外部から参照された一次情報を継続して更新する。
- 権威を自称せず、実績、継続性、第三者の認識によって示す。

## 6.5 Trustworthiness

Trustworthinessは、他の3要素を支える中心とする。

- 会社名、所在地、代表者、Contactを明確にする。
- ContentのAuthor、Reviewer、公開日、更新日を必要に応じて示す。
- 主張、数字、引用、成果にSourceと条件を持たせる。
- Serviceの対象、範囲、Price、制約、対象外を誠実に示す。
- Privacy、問い合わせ後の流れ、訂正方法を明確にする。
- 古い情報を放置せず、重要な訂正を隠さない。
- 広告、提携、Clientとの関係が判断へ影響する場合は明示する。

## 6.6 Who・How・Why

Contentごとに、次の3点を確認する。

| 観点 | 確認内容 |
| --- | --- |
| Who | 誰が作成またはReviewし、そのThemeにどのような関係があるか |
| How | どの経験、調査、取材、Data、Processに基づいて作られたか |
| Why | 検索流入の獲得だけでなく、誰のどの課題を解決するためか |

すべてのページへ同じ形式のAuthor欄を付けることを目的にしない。Newsは発表主体、Projectは担当者と支援範囲、ColumnはAuthorとReviewerなど、Contentの性質に合う責任情報を示す。

## 6.7 Page Type別の表現

| Page Type | 重視するE-E-A-T |
| --- | --- |
| Home | 事業領域、主要Service、代表Project、Companyへの一貫した接続 |
| Service | 専門性、対象、支援範囲、Process、関連Project |
| Project | 一次経験、担当範囲、判断、成果条件、掲載許可 |
| About / Company | 運営主体、代表者、沿革、所在地、公式情報 |
| Column / Guide | Author、Reviewer、Source、独自知見、更新日 |
| FAQ | 実際の疑問、正確な回答、関連Service、責任ある更新 |
| News | 発表主体、日付、事実、関連する公式情報 |

# 7. AI Search

## 7.1 基本方針

AI Search向けに別のContentを量産しない。検索ユーザーにとって有用で、正確で、一次情報を含むContentを整備することを基本とする。

Googleは、AI OverviewsやAI Modeへの掲載に特別なSEO要件や専用の最適化は不要と説明している。従来のPeople-firstなSEO方針と技術要件を満たすことが前提であり、条件を満たしても表示や引用は保証されない。詳細は[Google Search CentralのAI機能とWebsite](https://developers.google.com/search/docs/appearance/ai-features?hl=ja)を参照する。

## 7.2 一次情報と独自性

AIが既存情報を要約できるほど、一般論だけのContentは差別化しにくくなる。ジムグリットが情報源となれる内容を優先する。

- 自社Projectの背景、判断、Process、成果
- 代表者や担当者の専門的見解
- 自社で整理したData、Framework、比較軸
- 現場写真、資料、調査、検証方法
- 公式な会社、Service、Event、提携の情報
- 実務から得た制約、失敗、改善

独自性は奇抜さではなく、情報の出所と経験がジムグリットにあることを意味する。

## 7.3 理解・参照しやすい情報構造

AIに引用されること自体を目的にせず、人と検索システムが内容、責任、根拠を誤解しにくい構造をつくる。

- 一つのSectionで一つのQuestionまたは論点を扱う。
- 結論を先に示し、その後に理由、条件、Evidenceを置く。
- 固有名詞、Service名、数字、単位、期間を曖昧にしない。
- 主張とSource、Project、Authorを近い文脈で関連付ける。
- 定義、手順、比較、条件を必要に応じて整理する。
- 関連する補足ページへ文脈のあるInternal Linkを設ける。
- 内容が変わった場合に更新日を示し、形式的な日付変更をしない。

文章とページ内の情報順序はContent Guideに従う。

## 7.4 FAQ

FAQは、AI Search向けにQuestionを量産する場所ではない。問い合わせ、商談、検索行動で実際に確認された疑問を優先する。

- 一つのQuestionへ、最初に明確なAnswerを示す。
- 条件や例外がある場合は省略しない。
- Service、Project、Companyの情報と矛盾させない。
- 詳細が必要な場合は、一次情報を持つ関連ページへつなぐ。
- 同じQuestionを複数ページへ重複掲載しない。

## 7.5 Authorと更新情報

- 専門的なContentには、AuthorまたはReviewerと専門領域を示す。
- Author Profileから、関連するExperience、Project、資格を確認できるようにする。
- 公開日と更新日を区別し、実質的に更新した場合だけ更新日を変更する。
- 古い情報には、現在も有効か、終了済みかを明示する。
- AIを大きく利用したContentでは、ユーザーが作成方法を知ることに合理的な意味がある場合、利用方法と人による確認を説明する。

## 7.6 AI利用時の品質

AIは調査、整理、校正、案出しを支援できるが、経験、事実、専門性、責任の代わりにはならない。

- AIが生成した事実、Source、数字、引用、固有名詞を人が確認する。
- Projectに存在しない経験、成果、Commentを生成しない。
- 他Siteを組み合わせただけのContentを公開しない。
- AuthorまたはReviewerが、主張と事業上の責任を確認する。
- 大量生成より、一つのQuestionへ十分な独自価値を提供することを優先する。
- Search Rankingの操作を目的とした自動量産を行わない。

価値を加えない大量生成は、作成手段にかかわらず問題となる。AI利用時も[Google SearchのSpam Policy](https://developers.google.com/search/docs/essentials/spam-policies)を遵守する。

## 7.7 AI Searchの評価

AI回答への表示や引用だけを主要KPIにしない。表示は変動し、Site側で完全に制御できないため、次を合わせて評価する。

- 複雑なQuestionに答えるページが増えているか。
- 一次情報と独自Evidenceが蓄積しているか。
- Brand名、代表者、Service、Projectの情報が一貫しているか。
- AI Searchや検索から質の高い訪問が生まれているか。
- 指名検索、再訪、関連ページ閲覧、問い合わせへつながっているか。
- 誤った要約や古い情報を見つけた際に、公式情報で訂正できるか。

# 8. SEO Governance

## 8.1 基本方針

SEOは公開時に完了する作業ではない。検索行動、事業、Service、Project、競合環境、検索機能が変化することを前提に、観察、改善、更新を継続する。

短期的な順位変動へ反応して方針を頻繁に変えず、ユーザー価値、事業との関係、Content品質を基準に判断する。

## 8.2 Ownerと管理対象

SEO全体と主要ThemeにはOwnerを設定する。最低限、次を管理する。

- TargetとSearch Intent
- Theme / Query群
- Target PageとSupporting Page
- 関連するService、Project、Company
- Author、Reviewer、Evidence、Source
- Internal Link
- 公開日、最終確認日、更新Trigger
- 主なKPIと改善履歴

管理方法は運用規模に合わせ、更新できない複雑な台帳を作らない。

## 8.3 定期確認

| 頻度の目安 | 確認内容 |
| --- | --- |
| 月次 | 検索表示、主要Query、流入、行動、Errorや急変 |
| 四半期 | Search Intent、重要ページ、Internal Link、重複・孤立ページ、更新候補 |
| 随時 | Service、Project、会社情報、検索機能、公式方針の変更 |

頻度は事業影響と運用体制に応じてProject側で調整する。数値を見ること自体を目的にせず、変化の理由と対応の必要性を判断する。

## 8.4 KPI

SEOは複数の指標で評価する。

| 観点 | 例 |
| --- | --- |
| Visibility | 対象Queryでの表示、検索結果上の存在感 |
| Discovery | 非指名検索からの適切な新規訪問 |
| Brand | 指名検索、代表者・Service・Project名の検索 |
| Engagement | 関連ページ閲覧、再訪、必要情報への到達 |
| Business | 問い合わせ、相談、採用など目的行動への寄与 |
| Quality | 情報の正確性、独自性、更新性、孤立・重複の解消 |

順位、流入、問い合わせの単一指標だけで判断しない。季節性、Brand施策、広告、Media掲載、Site変更など他の要因も確認する。

## 8.5 改善Cycle

1. DataとユーザーのQuestionを確認する。
2. 問題がIntent、Content、Evidence、Link、技術のどこにあるか分ける。
3. 改善仮説と対象ページを決める。
4. Content GuideとDevelopment Guideの責務に従って更新する。
5. 変更内容、理由、日付を記録する。
6. 十分な期間を置いて、ユーザー行動と検索上の変化を確認する。

順位低下だけを理由に、無関係なKeyword追加、記事量産、全面書き換えを行わない。原因が技術にある場合はDevelopment、情報構造や文章にある場合はContentへ正しく切り分ける。

## 8.6 Contentの整理

- 正確で有用なContentは、変更せず維持する判断も認める。
- 古い情報は更新し、終了済みの事実はその状態を明示する。
- 同じIntentの重複ページは統合または役割を分ける。
- 事業との関係がなく、現在価値がなく、誤解を生むページは公開終了を検討する。
- Siteを新しく見せるためだけに、古いContentを一括削除または更新しない。
- 削除やURL変更の技術対応はDevelopment Guideに従う。

## 8.7 CommonとProject

- Commonには、複数案件で再利用できるSEO戦略だけを記載する。
- ジムグリット固有のTarget、Query、Theme、競合、KPI、Content計画はProject仕様で管理する。
- Project固有の成功例を、検証なしにCommonへ追加しない。
- 複数案件で有効性が確認された方針は、Reviewを経てCommonへの追加を検討する。
- CommonとProjectが矛盾する場合は、承認済みProject仕様を優先し、差異と理由を記録する。

## 8.8 Version管理

- 本ドキュメントを変更した場合はVersionを更新する。
- 変更履歴は `CHANGELOG.md` で管理する。
- 変更日、変更者、変更内容、変更理由、影響範囲を記録する。
- ProjectのSEO方針、Target Page、主要Theme、KPIの重要な変更も履歴を残す。
- 検索機能や外部方針の変化だけで即時変更せず、公式情報と実案件への影響を確認する。
- AI Searchを含む新しい知見は、再現性と継続性を確認してから標準へ反映する。
