# 0. About this Document

## 0.1 目的

本ドキュメントは、`02_Design_Principles.md` で定義した判断原則を、再利用可能なUIの見せ方へ落とし込むための標準である。

担当するデザイナー、Codex、開発者が変わっても、色、文字、余白、部品、画像、アイコン、動きの品質を同じ基準で再現できる状態をつくる。

## 0.2 責務

本ドキュメントは、次のUI Systemだけを扱う。

- Design Language
- Design Tokens
- Color System
- Typography
- Layout System
- Component Library
- Iconography
- Motion System

責務の境界は、次のとおりとする。

| Document | 定義すること |
| --- | --- |
| `02_Design_Principles.md` | なぜそのデザインを選ぶか |
| `03_Design_System.md` | UIをどう見せるか |
| `04_Development_Guide.md` | UIをどう実装し、品質を担保するか |
| `05_Content_Guide.md` | 何を、どの順番で伝えるか |

ページ単位の情報構成、Hero、Service、Feature、Project、Price、FAQ、Article、News、CTA、ContactなどのContent Patternは `05_Content_Guide.md` で管理する。ResponsiveとAccessibilityの実装・検証基準は `04_Development_Guide.md` で管理する。

## 0.3 適用と運用

Commonには再利用できるUIの役割と基準だけを置く。ブランド固有の値や承認済みデザインによる例外はProject側で管理し、CommonのToken名とComponentの役割は維持する。

新しいルールは、実案件で再利用性と有効性が確認された後に追加する。Versionと変更履歴の詳細は `00_PROJECT.md`、`04_Development_Guide.md`、`CHANGELOG.md` に従う。

# 1. Design Language

ジムグリットのUIは、情報が丁寧に整理され、静かで、信頼できることを基本とする。派手さや瞬間的な新しさではなく、内容が自然に理解でき、長く使っても古びにくい見せ方を選ぶ。

| キーワード | UIとしての意味 |
| --- | --- |
| Quiet | 表現同士を競わせず、情報に集中できる静けさがある |
| Premium | 装飾ではなく、精度、余白、一貫性から上質さが伝わる |
| Minimal | 必要な要素に絞られ、各要素の役割が明確である |
| Human | 無機質にならず、人や実際の体験を受け止められる |
| Functional | 見た目が理解、比較、判断、操作を支えている |
| Trustworthy | 情報と操作の状態が明快で、安心して使える |
| Timeless | 一時的な流行に依存せず、継続して使用できる |

`02_Design_Principles.md` の **Quiet Confidence** をUIへ反映し、一つの画面内で強く見せる要素を限定する。主情報、補足情報、操作の入口を分け、色、装飾、画像、動きが互いに競合しない状態を保つ。

情報が弱く見える場合は装飾を足す前に、文字の階層、余白、整列、まとまりを見直す。

# 2. Design Tokens

Design Tokenは、UIの見た目を一貫して再利用するための最小単位である。画面ごとの個別値ではなく、役割と段階を持つ共通体系として管理する。

| カテゴリ | 定義する内容 | 基準 |
| --- | --- | --- |
| Color | 文字、背景、境界、操作、状態に使う色 | 色名ではなく役割で区別する |
| Typography | 大きさ、太さ、行間、字間、用途 | 情報階層に対応する段階を限定する |
| Spacing | 要素内、要素間、まとまり間の余白 | 8pt Gridを基本に関係性を示す |
| Radius | 操作要素、容器、画像の角 | 少数の段階で統一する |
| Shadow | 浮き上がりと重なり | 境界や余白で表せない場合に限定する |
| Border | 区切り、輪郭、状態の境界 | 情報より目立たない強さにする |
| Motion | 状態変化と切り替えの動き | 目的と速度を少数の段階にする |

## 2.1 Token Naming Rule

Token名は `カテゴリ.役割` または `カテゴリ.段階` を基本とし、英小文字とピリオドで統一する。

| カテゴリ | 標準的な命名 |
| --- | --- |
| Color | `color.primary`、`color.secondary`、`color.background` |
| Typography | `typography.display`、`typography.h1`、`typography.body` |
| Spacing | `space.4`、`space.8`、`space.16`、`space.24`、`space.32` |
| Radius | `radius.sm`、`radius.md`、`radius.lg` |
| Shadow | `shadow.sm`、`shadow.md` |
| Border | `border.default`、`border.strong` |
| Motion | `motion.fast`、`motion.normal`、`motion.slow` |

- 見た目の特徴、ページ名、ブランド名をToken名に含めない。
- 同じ役割には同じToken名を使う。
- Projectで値を変更してもToken名は維持する。
- 既存の段階で表現できる場合は、新しいTokenを追加しない。

# 3. Color System

色は雰囲気を足すためではなく、情報の階層、操作、状態、境界を分かりやすくするために使う。Neutralと余白を基調にし、強い色は明確な役割を持つ場所に限定する。

| 役割 | 主な使用場面 | 注意点 |
| --- | --- | --- |
| Primary | 主要操作、重要な選択状態、ブランドの要所 | 多用せず、重要度を保つ |
| Secondary | 補助操作、カテゴリ、第二階層 | Primaryと競合させない |
| Accent | 限定的な注目や識別 | 装飾目的で反復しない |
| Neutral | 本文、見出し、補足、アイコン | 段階を増やしすぎない |
| Background | ページ全体の土台 | 清潔感と読みやすさを保つ |
| Surface | Card、Form、独立した操作領域 | 背景との差を色だけに頼らない |
| Border | 入力領域、区切り、輪郭 | 情報より目立たせない |
| Success | 正常な完了、利用可能 | 肯定的な状態に限定する |
| Warning | 注意、確認待ち、制約 | Errorと混同させない |
| Error | 入力不備、失敗、修正が必要な状態 | 修正対象を明確にする |

## 3.1 Default Color Tokens

ブランド固有の指定がない場合は、次のDefaultを出発点とする。HEXではなくTokenの役割を優先する。

| Token | Default | 役割 |
| --- | --- | --- |
| `color.primary` | `#1C2A35` | 中心的な識別と主要操作 |
| `color.secondary` | `#5F6B76` | 補助情報と補助操作 |
| `color.accent` | `#8A7050` | 限定的な注目と識別 |
| `color.neutral` | `#475467` | 文字、補足、図形の基調 |
| `color.background` | `#F7F7F5` | ページ全体の土台 |
| `color.surface` | `#FFFFFF` | 情報と操作のまとまり |
| `color.border` | `#D8DDE3` | 境界と区切り |
| `color.success` | `#287A55` | 正常な完了と肯定状態 |
| `color.warning` | `#A66A17` | 注意と確認が必要な状態 |
| `color.error` | `#B54747` | 問題と修正が必要な状態 |

## 3.2 Project Override

CommonではColor Tokenの役割だけを固定する。ブランドカラーはProjectごとに自由に上書きできるが、`color.primary` などのToken名は変更しない。

ブランド固有の色と値はProject側で管理する。上書き後も、各Tokenの役割、UI内の強弱、Project内の一貫性を維持する。

## 3.3 使用ルール

- 一つの画面で主役となる色を限定する。
- 主要操作や状態表示の色を、通常の装飾へ流用しない。
- 背景色の切り替えには、情報のまとまりを示す役割を持たせる。
- 鮮やかなブランドカラーは、使用面積を抑えて全体の静けさを保つ。

# 4. Typography

タイポグラフィは、読む順番と情報の関係を可視化する。日本語の読みやすさを優先し、装飾性の高い書体や、意味のないサイズ差・太さを増やさない。

| 種別 | 役割 | 主な使用場所 | 注意点 |
| --- | --- | --- | --- |
| Display | 最大階層と第一印象 | Homeの導入、特別な主題 | 短い語句に限定する |
| H1 | ページ全体の主題 | 各ページの先頭 | 原則一つとする |
| H2 | 大きな内容のまとまり | 主要Section | 前後の関係を明確にする |
| H3 | Section内の下位区分 | Card群、詳細項目 | H2との階層を保つ |
| Lead | 主題を補足する導入 | 主題やSectionの直下 | 見出しの反復にしない |
| Body | 継続して読む基本書式 | 説明、記事、詳細 | 行長と行間を整える |
| Small | 重要度の低い補足 | 注記、補足、メタ情報 | 主要情報の代用にしない |
| Caption | 画像や図表の補足 | 写真説明、図表、出典 | 対象の近くに置く |

## 4.1 組み方

- 見出しは大きさだけでなく、余白と整列でも階層を示す。
- 本文は行を詰めすぎず、長く読める行長と行間を保つ。
- 一画面で使うサイズと太さを必要最小限にする。
- 強調は太さ、位置、余白を優先し、色や装飾を多用しない。
- 英字は補助的に使い、日本語の理解を妨げない。
- 複数行の本文は、読み始めが揃う配置を基本とする。

# 5. Layout System

レイアウトは、情報の順番、まとまり、重要度を空間で示す。8pt Gridを基本とし、例外的な微調整は視覚的な整合に必要な場合に限定する。

| 項目 | 定義 |
| --- | --- |
| Container | 主要情報を保持し、左右の基準線を揃える共通領域 |
| Grid | 配置と余白の基準。8pt単位で一貫したリズムをつくる |
| Column | 関連情報を横方向に整理する単位 |
| Section | 一つの表示目的を持つ画面上のまとまり |
| Maximum Width | 文章や要素が広がりすぎることを防ぐ上限 |
| Whitespace | 情報の関係、分離、集中を示す空間 |

## 5.1 Spacing

- 関係の近い要素は近づけ、役割の異なる要素は離す。
- 同じ階層の要素は、位置、幅、内側余白、外側余白を揃える。
- 要素内、要素間、まとまり間、Section間の順に余白を広げる。
- 空間を埋めるために要素を増やさない。
- 広い表示領域でも本文を過度に横へ広げない。

## 5.2 Grid and Column

- Column数ではなく、比較のしやすさと読む順番を基準にする。
- 同列の要素は、画像、見出し、本文、操作の基準線を揃える。
- 情報量が異なる要素を、見た目だけを理由に同じ高さへ強制しない。
- 全幅の色面や画像は、画面上の明確な転換に必要な場合だけ使う。

## 5.3 Image Presentation

写真の選定思想と世界観は `02_Design_Principles.md` で定義する。本章では、UI上の表示だけを扱う。

- 同じ一覧内の画像は、アスペクト比と表示領域を統一する。
- 用途ごとに標準比率を決め、画面ごとの任意な比率を増やさない。
- トリミング後も、人物の顔、動作、サービスの文脈を欠損させない。
- 一覧では被写体の大きさと視点を揃え、項目間の比較を妨げない。
- Hero画像は主題を支える一枚を基本とし、複数画像を同じ強さで競合させない。
- 文字を画像上に置く場合は、被写体と重ならない表示領域を確保する。
- 画像の形状、Radius、Caption位置を同一用途で統一する。

# 6. Component Library

Componentは、同じ役割を同じ見た目と状態で伝えるUIの最小単位である。ページやContentの構成は定義せず、再利用可能な表示と操作の単位だけを扱う。

| Component | UI上の目的 | 主な使用場面 | 使わない場面 |
| --- | --- | --- | --- |
| Button | 実行可能な操作を明確に示す | 送信、確定、画面遷移 | 単なる強調、行動が曖昧な場合 |
| Card | 同種の情報を独立した単位で表示する | 一覧、比較、選択 | 連続して読む本文、構造が異なる情報 |
| Input | 一つの情報を入力・選択する状態を示す | 文字入力、選択、自由記述 | Labelや状態が判断できない場合 |
| Form | 複数のInputを一つの操作目的にまとめる | 問い合わせ、申込み | 目的以上の情報を求める場合 |
| Navigation | 現在地と主要な移動先を示す | Global、Local、補助Navigation | 移動先の羅列、階層が不明な場合 |
| Header | ブランド識別と主要Navigationを保持する | 画面上部の共通領域 | 告知や操作を過密に配置する場合 |
| Footer | 補助Navigationと基礎情報を整理する | 画面末尾の共通領域 | 主要情報をFooterだけに置く場合 |
| Badge | 短い属性や状態を識別する | Category、状態、補助Label | 長文、主要見出し、装飾だけの表示 |
| Accordion | 詳細の開閉状態を示す | 補足、条件、長い詳細 | 全員が常に読むべき情報 |
| Modal | 現在の画面上で短い確認や操作を行う | 確認、重要な補助操作 | 長文閲覧、通常の画面遷移で足りる場合 |

## 6.1 Content Display Components

Table、List、QuoteはContentの内容や順序を定義しない。03では、与えられたContentを一貫して表示するUIとして扱う。

| Component | 表示上の基準 |
| --- | --- |
| Table | 見出し、行、列、区切りの関係を揃え、比較軸を視覚的に保つ |
| List | 同列または順序の違いが分かる記号、字下げ、間隔を統一する |
| Quote | 本文との違い、引用範囲、出所の位置関係を一貫させる |

## 6.2 共通ルール

- 一つのComponentに複数の主要目的を持たせない。
- Default、Hover、Active、Disabled、Errorなど必要な状態を同じ体系で表す。
- 状態差は色、境界、文字、形の組み合わせで明確にする。
- 同じComponentは、余白、Radius、文字階層、アイコン位置を揃える。
- Variantは役割の違いを説明できる場合だけ追加する。
- Card内の情報構成やCTAのタイミングは `05_Content_Guide.md` に従う。

# 7. Iconography

アイコンは、操作や情報の意味を短時間で理解するための補助として使う。線アイコンを基本とし、同一Project内では線の太さ、角、視覚的な大きさ、余白を揃える。

- 一目で意味が伝わる、一般的でシンプルな形を選ぶ。
- 原則としてテキストと併用し、アイコンだけに意味を依存しない。
- 操作、状態、カテゴリなど明確な役割がある場合に使う。
- 同じ意味には同じアイコンを使う。
- 小さな表示でも判別できる複雑さに抑える。
- 線と塗り、異なるテイストを無秩序に混在させない。
- 装飾や余白埋めを目的として使わない。

# 8. Motion System

Motionは、`02_Design_Principles.md` の **Quiet Confidence** をUIの状態変化へ落とし込む。自然で、静かで、滑らかであり、ユーザーの理解や操作を邪魔しないことを目的とする。

| 種別 | 目的 | 表現の基準 |
| --- | --- | --- |
| Hover | 操作可能であることを伝える | 色、境界、位置の変化を小さくする |
| Fade | 表示・非表示を穏やかに伝える | 内容を待たせず、短く滑らかに変化させる |
| Scroll | 読む流れや現在位置を補助する | 表示の遅れや視線の強制を生まない |
| Loading | 待機中であることを伝える | 状況を明示し、過度に目立つ反復を避ける |
| Transition | 状態や画面の関係を伝える | 移動元と移動先が理解できる変化にする |

## 8.1 Motion Rules

- `motion.fast`、`motion.normal`、`motion.slow` の少数段階で速度を統一する。
- 同じ操作と状態変化には、同じMotionを使う。
- 同時に多数の要素を動かさない。
- 長い遅延、強い視差、自動的な連続移動、過度な拡大縮小を行わない。
- 情報を読む、選ぶ、戻る、送る行動をMotionで妨げない。
- 派手な演出や「見せるためのアニメーション」は行わない。
- Motionがなくても、内容と状態を理解できる見せ方を保つ。
